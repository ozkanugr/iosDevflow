# Brainstorming Question Bank

Comprehensive question sets for all 14 brainstorming modes. Use these as foundations, adapting based on project type and user responses.

---

## Mode: 5w1h

### WHO - Stakeholders & Users

1. Who is the primary user of this application?
2. What is their technical skill level?
3. What demographics define your target audience?
4. Who are secondary/indirect users?
5. Who are the project stakeholders (product owner, designers, QA)?
6. How large is the development team?
7. What experience level does the team have with the chosen technology stack?
8. Are there accessibility requirements for specific user groups?
9. Who makes the purchasing decision vs. who uses the product?
10. Who influences the user's choice?

### WHAT - Product Definition

1. Describe the core functionality in one sentence.
2. What are the top 3 problems this solves?
3. What existing apps/tools do users currently use for this?
4. What makes your solution different from existing alternatives?
5. What data types does the application handle?
6. What third-party services or APIs must it integrate with?
7. What are the non-negotiable features for v1.0?
8. What features are you explicitly deferring?
9. Does the app need to work offline?
10. What notification/communication features are needed?

### WHEN - Timeline & Phases

1. What is the target MVP completion date?
2. What is the full v1.0 release target?
3. Are there external deadlines (events, regulations, contracts)?
4. What is the desired release cadence after v1.0?
5. Is there a beta/testing period planned?
6. Are there seasonal or market timing considerations?

### WHERE - Distribution & Environment

1. Which platforms will this target?
2. What is the minimum version/environment requirement?
3. Where will this be distributed?
4. Where does the backend/API infrastructure live?
5. Where is user data stored? (On-device, cloud, custom server)
6. What geographic regions must be supported?
7. Are there data residency/sovereignty requirements?

### WHY - Purpose & Value

1. Why does this need to exist?
2. What business model supports this? (Free, Freemium, Paid, Subscription)
3. What KPIs define success for this project?
4. Why is now the right time to build this?
5. What happens if this project is not built?
6. What is the competitive advantage?

### HOW - Technical Approach

1. What architecture pattern should be used?
2. What UI framework or approach?
3. How will the team handle code reviews?
4. How will CI/CD be configured?
5. How will secrets/credentials be managed?
6. How will performance be monitored?
7. How will error/crash reporting work?
8. What testing strategy? (Unit, Integration, UI, E2E)
9. How will localization be handled?
10. How will accessibility compliance be verified?

## Mode: design-thinking

### Empathize

1. What does a typical day look like for your user?
2. What frustrates users about current solutions?
3. What workarounds do users currently employ?
4. What would users never give up from current solutions?
5. What do users complain about most?
6. What tasks take longer than users expect?
7. What makes users feel accomplished?
8. What makes users feel frustrated?

### Define

1. What is the single biggest problem to solve?
2. What insight about users surprised you?
3. How would you complete: "[User] needs a way to [X] because [Y]"?
4. What assumptions are we making about the problem?
5. What would "success" look like from the user's perspective?

### Ideate

1. What's the simplest possible solution?
2. What's the most innovative solution (ignore constraints)?
3. How would [competitor] solve this?
4. How would a completely different industry solve this?
5. What if we had unlimited budget?
6. What if we had to launch in one week?

### Prototype

1. What's the minimum interaction needed to validate?
2. What screens are absolutely essential?
3. What's the critical user flow?
4. What data is needed on each screen?
5. What actions can users take?

### Test

1. What would prove our hypothesis wrong?
2. What behavior would indicate success?
3. What questions should we ask testers?
4. What metrics would tell us it's working?
5. What's the smallest group we can test with?

## Follow-Up Question Patterns

### When answers are vague

- "Can you give me a specific example?"
- "What would that look like in practice?"
- "If you had to choose just one, which would it be?"
- "How would you measure success for that?"

### When answers reveal complexity

- "That sounds like multiple features. Shall we break it down?"
- "What's the simplest version that would still be valuable?"
- "Are there dependencies between those requirements?"
- "Which part is most critical?"

### When answers conflict

- "Earlier you mentioned {X}, but this seems different from {Y}. How should we reconcile?"
- "Given the timeline, is this realistic for v1.0?"
- "This might impact {other requirement}. What's the priority?"
- "Can we do both, or must we choose?"

### When user says "I don't know"

- "What do similar products do?"
- "If you had to guess, what would you say?"
- "Let me offer some options: [A], [B], or [C]?"
- "Should we mark this as an open question to research?"

## Depth Indicators

| User Response | Recommended Action |
|---------------|-------------------|
| Detailed, specific answer | Move to next question |
| Hesitant or vague | Probe deeper with follow-ups |
| "I don't know" | Offer 2-4 concrete options |
| "Not applicable" | Acknowledge and move on |
| Contradicts earlier answer | Surface conflict and reconcile |
| Enthusiasm | Explore deeper, may reveal priorities |
| Frustration | May indicate critical pain point |
